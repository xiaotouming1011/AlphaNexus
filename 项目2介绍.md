# 项目2介绍：从混合RAG到图RAG的智能问答系统

我做了一个从混合 RAG 演进到图 RAG 的智能问答项目，目标是把“语义检索能力”和“关系推理能力”放到同一套可落地的工程里。

在混合 RAG 阶段，我以 Markdown 语料为核心数据源，先做结构化清洗，再基于标题层级（`# / ## / ###`）做语义友好的分块。检索层采用 Dense 向量召回 + Sparse 关键词召回（BM25）的双通道方案，并通过 RRF 做融合，提升召回覆盖和稳定性，减少单一检索器造成的漏召回。

在图 RAG 阶段，我将系统升级为“向量检索 + 知识图谱”协同架构：用 Neo4j 承载实体关系网络，用 Milvus（HNSW）做高效语义检索，并引入查询路由策略，按问题复杂度在传统检索、图检索或组合检索之间动态选择。针对复杂问题，系统支持多跳遍历、子图提取和关系链推理，能够回答只靠相似度检索难以覆盖的关系型问题。

## 技术选型说明（面试可讲）

- 为什么选 BGE（中文 Embedding）
  项目主要处理中文问答场景，BGE 在中文语义表示上的鲁棒性更好，对同义改写、口语提问和长尾表达的召回效果更稳定，能直接提升 Dense 检索质量。

- 为什么做 Dense + Sparse 混合检索
  Dense 擅长语义匹配，Sparse 擅长关键词精准命中（菜名、术语、实体名）。两者互补，能在开放问答和实体问答里同时保证覆盖率和准确率。

- 为什么用 RRF 融合
  RRF 对不同检索器分数尺度不敏感，不依赖复杂归一化，工程实现简单且效果稳定，适合快速落地和持续迭代。

- 为什么引入图 RAG（Neo4j）
  当问题涉及多实体关系、路径依赖、因果或组合推理时，纯向量检索很难显式建模关系结构。图数据库能提供多跳检索和结构化推理能力，补齐复杂问答短板。

- 为什么 Milvus 选 HNSW
  在问答系统里需要兼顾低延迟和高召回，HNSW 通常是较稳的工程折中；同时支持持续增量写入，适合知识库不断更新的场景。

## 我在这个项目里最想体现的能力

不仅能把 RAG 跑通，还能从数据处理、检索策略、图推理到系统可运行性做全链路设计与落地，把“可用”进一步推进到“可解释、可扩展、可演示”。
