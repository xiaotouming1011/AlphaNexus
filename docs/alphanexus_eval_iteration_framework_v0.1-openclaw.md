# AlphaNexus 评测与迭代框架（OpenClaw 版本 v0.1）

> 生成时间：2026-03-04（GMT+8）
> 依据：GitHub 仓库 `xiaotouming1011/AlphaNexus` + 线上服务 `https://115.191.38.43`

## 0. 项目回顾（面试开场可用）

AlphaNexus 是一个多智能体投研系统，核心链路是：
市场/社媒/新闻/基本面分析员 -> Bull/Bear 投资辩论 -> Trader 交易计划 -> Aggressive/Conservative/Neutral 风控辩论 -> Risk Judge 最终 Buy/Sell/Hold。

当前线上验证到的事实：
- `https://115.191.38.43/api/health` 返回 `{"status":"ok"}`。
- `https://115.191.38.43/api/portfolio/health` 返回 `{"status":"ok","service":"portfolio"}`。
- HTTP 80 端口可连通，但 `http://115.191.38.43` 请求超时，线上入口应以 HTTPS 为准。
- 首页是 React Landing（`Antinertia | V2 React Bits`），核心控制台在 `/console`。

## 1. 评估体系构建（主观 + 客观）

### 1.1 评估总框架（建议面试里先讲这个）

把评估拆成四层：
- `L1 过程质量`：每个 agent 是否按预期完成、辩论是否有效。
- `L2 结果质量`：结论是否有证据、是否逻辑一致、是否可执行。
- `L3 工程质量`：成功率、时延、稳定性、错误可恢复。
- `L4 业务质量`：用户是否信任、是否可复盘、是否能推动决策。

建议总分：
`TotalScore = 0.35 * 过程质量 + 0.35 * 结果质量 + 0.20 * 工程质量 + 0.10 * 业务质量`

### 1.2 主观维度（人工评审 Rubric，1-5 分）

1. `可解释性`：结论是否能追溯到来源与辩论过程。  
2. `逻辑一致性`：报告、辩论、最终结论是否前后一致。  
3. `完整性`：四类分析、两段辩论、最终结论是否齐全。  
4. `可操作性`：输出是否给出明确行动建议与风险边界。  
5. `可读性`：结构清晰、信息密度合适、用户能快速理解。

评审方法：
- 每个 case 由 2 人独立打分，取均值。
- 对分歧 >1 分的 case 做仲裁，沉淀“评分口径示例”。

### 1.3 客观维度（自动化指标）

1. `Run Success Rate`：成功运行占比（排除上游 API 全局故障）。  
2. `P50/P90 完成时长`：从请求到 final event 的时延。  
3. `首包时延`：SSE 首个事件返回时间。  
4. `来源覆盖率`：关键结论句中带来源证据的比例。  
5. `无来源断言率`：高置信断言但无来源支撑的比例。  
6. `辩论有效轮次`：`investment_debate_state.count` 与 `risk_debate_state.count`。  
7. `稳定性`：同输入多次运行，结论方向一致率。  
8. `风格隔离正确率`：risk_profile 只影响 `news_report_ui`，不污染主链路。

面试里可补一句：
“我会把主观评分和客观指标打通，形成一个 weekly scorecard，避免只靠感觉说系统变好了。”

## 2. 数据评测集：获取、梳理、持续更新

### 2.1 如何获取评测集（Data Acquisition）

建议三类数据并行：

1. `离线基准集`（可重复）：
- 覆盖典型场景：财报日、宏观事件日、行业突发事件、平静交易日。
- 每条包含：ticker、trade_date、外部事实锚点（新闻/公告/行情）。

2. `在线真实流量集`（最有价值）：
- 从线上 `/api/run` 与 `/api/run/stream` 的请求/响应抽样。
- 重点保留失败样本、超时样本、用户反馈负面样本。

3. `对抗/边界集`（专测鲁棒性）：
- 极端日期、冷门 ticker、数据缺失、输入歧义、中文英文混输。

### 2.2 如何梳理评测集（Data Curation）

每条样本统一结构（建议）：
- `case_id`
- `input`: ticker/trade_date/provider/data_vendor/risk_profile
- `expected_constraints`: 例如“必须给出风险提示”“必须有>=N条来源”
- `ground_truth_refs`: 外部事实链接
- `labels_process`: 完整性/一致性/可解释性
- `labels_result`: 结论合理性/可操作性
- `failure_tags`: timeout、hallucination、source_missing、logic_conflict
- `dataset_version`

切分策略：
- `smoke`：10-20 条，发布前 5 分钟跑完。
- `regression`：100+ 条，周更。
- `challenge`：困难样本，专测多跳推理与风控辩论。

### 2.3 上线后如何补充与替换更新

建议用“滚动窗口 + 版本化”：

1. 每天新增：
- 从线上抽取 badcase（至少 5-10 条）入 `candidate pool`。

2. 每周评审：
- 通过标注后，晋升到 `regression`；低价值旧样本淘汰。

3. 每月重平衡：
- 确保各场景分布均衡（行业、波动水平、事件类型、成功/失败比）。

4. 每次发版门禁：
- 新版本在 `regression` 上不能出现关键指标退化（例如成功率、来源覆盖率、P90 时延）。

## 3. 每天线上扒 badcase：定位原因 + 推动迭代

### 3.1 每日 badcase 流程（可讲成 SOP）

1. `自动筛`：按规则抓异常样本。  
2. `人工看`：按优先级选 Top N。  
3. `打标签`：统一 failure taxonomy。  
4. `定归因`：定位到数据/模型/Prompt/编排/UI。  
5. `立改动`：生成 issue 与验收标准。  
6. `回归验证`：修复后回放对应 case，防止复发。

### 3.2 常用异常抓取规则

- 请求超时或中断。
- `sources` 数量过低或无效链接比例过高。
- 关键模块输出为空（如 `market_report` 空）。
- 投资辩论/风控辩论轮次不足。
- 结论和风险偏好明显冲突（例如“非常保守”却给高杠杆建议）。

### 3.3 根因分析模板（面试好用）

- `输入层`：ticker/date/risk_profile 是否异常。
- `数据层`：数据源限流、缺失、字段漂移。
- `模型层`：provider 不稳定、模型漂移、超长上下文退化。
- `编排层`：状态机跳边错误、节点输出未传递。
- `产品层`：前端没展示、用户误解文案。

### 3.4 如何推动产品迭代

把 badcase 转成可执行项：
- `问题定义`：一句话描述现象和影响范围。
- `改动策略`：改 Prompt / 改编排 / 改 fallback / 改 UI 提示。
- `验收标准`：对应 case 通过 + 关键指标不退化。
- `复盘结论`：是否进入长期评测集。

## 4. QA 知识库构建：如何收集 QA 对

### 4.1 QA 来源（先把入口建全）

1. 用户真实提问与客服对话。  
2. GitHub Issues / PR 评论。  
3. 线上错误日志（HTTP 4xx/5xx、timeout、rate limit）。  
4. 产品文档（README、PRD、部署文档）。

### 4.2 QA 卡片结构（建议）

- `question`：用户会怎么问（口语化）。
- `answer_short`：一句话直答。
- `answer_detail`：分步骤解释。
- `applicable_scope`：适用版本/模块。
- `source_of_truth`：对应代码或文档链接。
- `last_verified_at`：最后核验时间。
- `owner`：负责人。

### 4.3 QA 对生成与维护流程

1. `收集`：每天汇总新增问题。  
2. `去重聚类`：同义问题合并。  
3. `写答案`：先短答再详答，避免长篇空话。  
4. `技术核验`：由工程侧确认可执行。  
5. `发布`：进入 FAQ/内部知识库。  
6. `更新`：发版后批量核验失效答案并替换。

### 4.4 QA 知识库成效指标

- `命中率`：用户问题是否能在知识库命中。
- `一次解决率`：首答即解决比例。
- `人工转接率`：无法回答需要人工介入比例。
- `过期率`：答案与现网不一致比例。

## 5. 面试可直接说的 90 秒总结（可背）

我把 AlphaNexus 当成一个“可评测、可迭代”的多智能体产品来做，而不是 Demo。  
第一，我设计了主观+客观双轨评估体系，主观看可解释性和逻辑一致性，客观看成功率、时延、来源覆盖率、辩论轮次与稳定性。  
第二，我把评测集分成离线基准、线上真实流量和对抗边界集，并用版本化管理，保证每次发版有回归门禁。  
第三，我每天从线上自动抓 badcase，按输入、数据、模型、编排、产品五层定位根因，再把问题转成可验收迭代项。  
第四，我搭了 QA 知识库流程，把用户问题、日志错误和文档沉淀成标准 QA 卡片，持续降低重复问题和人工成本。  

## 6. 你下一步最值得做的两件事

1. 先做 `weekly scorecard`（哪怕先手工），让“系统变好”有数字证据。  
2. 先建 `badcase -> regression` 晋升机制，这是你面试里最能体现 PM + AI 工程思维的抓手。
