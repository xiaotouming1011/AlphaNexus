# PRD：AlphaNexus 多智能体投研控制台（V1.0）

- 版本：2026-02-25
- 状态：Draft
- 适用范围：基于当前实现（`/Users/anji/TradingAgents_副本`）

## 1. 产品概述

AlphaNexus 是一个面向美股研究场景的多智能体投研平台。系统将“分析-辩论-风控-结论”串联为可视化流程，支持 Web 与 CLI 使用方式，目标是让用户在一个界面内完成从标的输入到最终交易建议的完整研究闭环。

## 2. 背景与问题

1. 信息来源分散，数据与结论不可追溯。
2. 单模型结论容易偏向单一路径，缺少反方校验。
3. 研究流程缺乏结构化沉淀，难复盘、难展示。
4. 海外市场信息获取门槛高，常依赖 VPN；且网站与信息源数量多、格式不一，难以高效汇总，难形成体系化研究。

AlphaNexus 通过多角色智能体分工与辩论机制，提升结论可解释性与过程透明度。通过 MCP 获取 API 规范，直接经 API 调用绕开 VPN 依赖，直达信息源头，并将海量高质量元数据与 LLM 能力集成。

## 3. 产品目标

1. 提供端到端投研流程：输入标的与日期后，自动完成分析、辩论、风控与结论输出。
2. 输出可追溯：每次运行保留来源链接、阶段性产物、辩论历史。
3. 支持个性化新闻视图：基于“投资风格”生成新闻展示版本（不影响辩论与总结主链路）。
4. 支持结果沉淀：支持 Markdown/JSON 下载，便于复盘与面试展示。

## 4. 非目标（Out of Scope）

1. 不提供自动实盘交易。
2. 不保证投资收益，不构成投资建议。
3. 不做券商下单对接（当前阶段）。

## 5. 目标用户

1. 求职/实习候选人：需要展示 AI + 金融 + 工程化能力。
2. 个人投资研究者：希望快速获得结构化研究报告。
3. AI 工程实践者：关注多智能体协作和流程编排。

## 6. 核心使用场景

1. 用户输入 `Ticker/公司名 + 交易日期(ET)`，选择单一分析模块，发起流式运行。
2. 用户选择 LLM 供应商与模型，复用服务端 API Key 或临时输入。
3. 用户选择“投资风格”（5 档），查看风格化新闻展示结果。
4. 用户查看辩论记录、来源列表、公司关系图谱，并导出结果。

## 7. 功能需求（Functional Requirements）

### FR-01 研究页输入与校验

1. 支持股票代码/公司名识别（含热门美股映射）。
2. 日期使用 ET 语义，允许休市日继续分析并提示。
3. 支持分析模块四选一：市场/社媒/新闻/基本面。

### FR-02 多模型与数据源配置

1. 支持 OpenAI、Anthropic、Google、xAI、OpenRouter、Ollama、DeepSeek。
2. 支持 `deep_think_llm` 与 `quick_think_llm` 配置。
3. 数据源支持 `yfinance` 与 `alpha_vantage`。
4. API Key 优先级：请求参数 > 服务端环境变量。

### FR-03 多智能体主链路

1. 分析员产出模块报告。
2. 投资辩论（Bull/Bear）多轮（adaptive/fixed，最少 2 轮）。
3. 交易员整合计划。
4. 风控辩论（Aggressive/Conservative/Neutral）多轮。
5. 输出最终交易结论。

### FR-04 投资风格新闻筛选

1. 提供 5 档投资风格：非常保守/比较保守/适中/比较激进/非常激进。
2. 风格参数随运行请求透传。
3. 仅影响新闻展示层。
   - 保持 `news_report` 中性供下游辩论与总结使用。
   - 生成 `news_report_ui` 作为前端展示。

### FR-05 可视化与流式反馈

1. `POST /api/run/stream` 以 SSE 返回状态、进度、报告、辩论、最终结果。
2. 前端展示阶段进度、模块输出、辩论气泡、Sources 与图谱。

### FR-06 报告与来源

1. 支持 Markdown/JSON 下载。
2. Sources 列表集中展示、支持外链跳转。
3. 支持公司关系图谱（影响路径、关系边）。

### FR-07 组合页

1. 支持组合价值与权重可视化。
2. 支持 Alpha Vantage 限流与缓存回退。

## 8. 非功能需求（NFR）

1. 可用性：核心运行链路成功率（剔除上游 API 故障）>=95%。
2. 时延目标。
   - 首个 SSE 事件 <=2s。
   - 首个分析报告 <=60s（取决于供应商）。
   - 完整结果 P90 <=240s。
3. 可观测性：记录每阶段状态、失败原因与供应商信息。
4. 安全性：前端不持久化用户密钥，默认走服务端环境变量。
5. 兼容性：支持本地端口访问，不依赖域名。

## 9. 关键指标（KPI）

1. 运行成功率（Run Success Rate）。
2. 平均完成时长与 P90 完成时长。
3. 来源覆盖率（报告中可追溯来源占比）。
4. 用户导出率（Markdown/JSON 下载占比）。
5. 新闻风格切换有效率（不同风格输出差异显著且无事实漂移）。

## 10. 风险与应对

1. 第三方模型不稳定或限流（如 Gemini 429、代理异常）。
   - 应对：重试、降级模型、提示切换供应商、环境变量与代理健康检查。
2. 来源链接失效或质量不均。
   - 应对：来源抽取规则增强与去重策略。
3. 推理耗时波动大。
   - 应对：优化 prompt 长度、分层模型调用、流式可视化缓解等待感。

## 11. 里程碑建议

1. M1（1 周）：PRD 冻结 + 指标埋点 + 异常分类。
2. M2（1-2 周）：稳定性优化（代理/重试/降级）+ 速度优化。
3. M3（1 周）：评测报告产出（前后对比、可量化指标）。
4. M4（1 周）：面试材料化（架构图、Demo 脚本、案例集）。

## 12. 验收标准

1. 用户可在本地端口完成一次完整运行并得到可下载结果。
2. 投资风格变更仅影响新闻展示，不影响辩论与最终结论链路输入。
3. 运行失败时返回可读错误，并可定位到供应商/网络/参数层。
4. Sources、辩论记录、图谱信息可在 UI 正常展示。
