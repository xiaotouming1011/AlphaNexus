# AlphaNexus 多智能体投研系统 Gemini 版评测与迭代框架 v0.1

## 1. 目标与范围

本框架旨在为 AlphaNexus 多智能体投研系统提供一套系统化、可执行的评测与迭代规范，以确保系统决策质量、稳定性及用户体验。

**核心目标：**

*   建立统一的评估标准和流程。
*   持续优化智能体的决策能力和系统稳定性。
*   构建可追溯、可量化的评测机制。
*   加速 badcase 的发现、归因与修复闭环。
*   沉淀系统运行经验与知识，提升团队效率。

**范围：**

本框架覆盖 AlphaNexus 系统从市场/社媒/新闻/基本面分析到多空辩论、交易计划、风控辩论直至最终 Buy/Sell/Hold 决策的全链路。重点关注各智能体间的协作、信息传递、决策质量、溯源能力（来源追溯）和用户体验（SSE 流式、risk_profile 影响新闻展示层）。

## 2. 评估体系

### 2.1 主观维度（定性评估）

通过人工复核和专家打分进行。

*   **决策逻辑清晰度：** 智能体决策过程是否可理解，理由是否充分。
    *   **KPI建议：** 专家评分 ≥ 4/5 (5分制)
*   **信息整合度：** 各智能体输出信息是否有效整合，无冗余或冲突。
    *   **KPI建议：** 专家评分 ≥ 4/5
*   **论证全面性：** 多空辩论环节是否充分考虑正反两方面，论据扎实。
    *   **KPI建议：** 专家评分 ≥ 3.5/5
*   **风控合理性：** 风控辩论对风险点的识别和应对建议是否合理。
    *   **KPI建议：** 专家评分 ≥ 4/5
*   **可解释性与溯源：** 决策是否能清晰追溯到原始数据源和智能体链条。
    *   **KPI建议：** 专家评分 ≥ 4.5/5
*   **用户体验（前端展示）：** SSE 流式更新是否流畅，risk_profile 对新闻展示的影响是否符合预期。
    *   **KPI建议：** 用户满意度 ≥ 80%

### 2.2 客观维度（定量评估）

通过自动化脚本和系统日志数据进行。

*   **决策准确率：** 最终 Buy/Sell/Hold 决策与真实市场表现（或专家标注）的一致性。
    *   **KPI建议：** 初步目标 ≥ 60%，长期目标 ≥ 75%
*   **信息完整度：** 决策所需关键信息（如价格、新闻、情绪）的覆盖率。
    *   **KPI建议：** ≥ 95%
*   **响应延迟 (Latency)：** 从输入到输出最终决策的平均耗时。
    *   **KPI建议：** 平均 ≤ 5秒 (针对实时分析场景)
*   **资源消耗：** CPU、内存、Token 使用量等。
    *   **KPI建议：** 保持在预算范围内，同比优化 ≥ 10%
*   **来源追溯成功率：** 随机抽检 N 个决策，验证其信息来源是否可追溯。
    *   **KPI建议：** ≥ 99%
*   **输出一致性：** 同一输入多次执行，关键输出是否保持一致。
    *   **KPI建议：** ≥ 90% (针对非随机性输出)
*   **Badcase 修复率：** 已发现 badcase 的修复比例。
    *   **KPI建议：** ≥ 90%

### 2.3 评分权重

| 维度     | 具体指标             | 权重 |
| :------- | :------------------- | :--- |
| **主观** | 决策逻辑清晰度       | 15%  |
|          | 信息整合度           | 10%  |
|          | 论证全面性           | 10%  |
|          | 风控合理性           | 10%  |
|          | 可解释性与溯源       | 5%   |
|          | 用户体验             | 5%   |
| **客观** | 决策准确率           | 20%  |
|          | 信息完整度           | 5%   |
|          | 响应延迟             | 5%   |
|          | 资源消耗             | 5%   |
|          | 来源追溯成功率       | 5%   |
|          | 输出一致性           | 5%   |
| **总计** |                      | 100% |

## 3. 评测集建设与版本化

评测集是系统迭代的基石，需持续建设和版本化管理。

### 3.1 获取

*   **历史数据回测：** 使用历史市场数据、新闻、社交媒体情绪等，模拟过往交易日情境。
*   **真实用户行为：** 匿名化收集线上用户输入，作为评测集的补充。
*   **专家生成：** 邀请金融专家、研究员手动构造特定场景（如突发事件、政策变动、财报季等）。
*   **对抗性样本生成：** 尝试生成极端或容易混淆的输入，测试系统鲁棒性。

### 3.2 清洗

*   **数据脱敏：** 移除敏感信息。
*   **格式统一：** 确保输入数据的格式符合系统要求。
*   **去重与冗余：** 清理重复或低价值样本。
*   **专家标注：** 对部分评测集进行人工标注，作为“黄金标准”。

### 3.3 分层

根据复杂度和场景将评测集分层管理。

*   **冒烟测试集 (Smoke Test Set)：** 少量关键、高频场景，用于快速验证核心功能。
*   **回归测试集 (Regression Test Set)：** 覆盖主要业务流程及历史 badcase 修复后的验证。
*   **性能测试集 (Performance Test Set)：** 针对高并发、大数据量场景，评估系统性能。
*   **极端场景测试集 (Edge Case Test Set)：** 市场崩溃、黑天鹅事件、数据缺失等异常情况。
*   **多智能体协作测试集：** 专门用于评估智能体间信息流转、辩论及决策的质量。

### 3.4 更新与版本化

*   **定期更新：** 每月或每季度对评测集进行更新，加入新的市场热点、badcase 等。
*   **版本管理：** 对评测集进行版本控制，确保不同版本的系统在相同基准上进行评估。
*   **Badcase 融入：** 修复后的 badcase 应被清洗并纳入回归测试集。

## 4. 线上 badcase 运营闭环

构建高效的 badcase 闭环机制，快速响应和解决生产问题。

### 4.1 发现

*   **用户反馈：** 通过用户界面、客服渠道收集问题报告。
*   **日志监控：** 实时监控系统日志，识别异常、错误或低质量决策。
*   **自动化异常检测：** 基于规则或机器学习模型，自动识别与历史表现偏差较大的决策。
*   **人工巡检：** 定期抽样检查系统输出，进行人工评估。

### 4.2 归因

当发现 badcase 后，需进行多维度归因分析：

*   **智能体层面：** 哪个智能体（分析师、辩论者、研究员、交易员、风控经理）的输出导致问题？是逻辑缺陷、Prompt 问题还是数据处理问题？
*   **工具层面：** 调用的外部工具（如 Alpha Vantage、YFinance）是否返回错误或异常数据？
*   **LLM 层面：** LLM 的理解偏差、幻觉或生成内容与预期不符？
*   **数据层面：** 输入数据是否错误、缺失、延迟或格式不正确？
*   **系统层面：** 智能体编排、状态管理、内存机制是否存在问题？
*   **Prompt 工程：** Prompt 定义是否模糊、存在歧义或引导不当？

### 4.3 修复

*   **优先级评估：** 根据影响范围、用户量、紧急程度对 badcase 进行优先级排序。
*   **代码修复：** 修正智能体逻辑、工具调用、API 集成等。
*   **Prompt 优化：** 迭代 Prompt，使其更清晰、鲁棒。
*   **数据修正：** 清洗或补充缺失/错误数据。
*   **模型微调/切换：** 考虑是否需要对底层 LLM 进行微调或尝试其他模型。

### 4.4 回归

*   **Badcase 特定回归：** 针对已修复的 badcase，确保其不再复现。
*   **回归测试集运行：** 在修复后，运行相关的回归测试集，确保修复不引入新的问题。
*   **灰度发布：** 对于关键修复，可考虑小范围灰度发布，观察线上表现。

## 5. QA 知识库体系

建立结构化的 QA 知识库，沉淀经验，提升团队解决问题的效率。

### 5.1 收集

*   **Badcase 归档：** 将已解决的 badcase、归因分析和修复方案录入知识库。
*   **专家经验：** 记录资深专家对特定市场事件、复杂场景的处理思路和决策逻辑。
*   **LLM 行为洞察：** 记录不同 LLM 在特定 Prompt 下的典型行为、优缺点。
*   **智能体设计文档：** 各智能体的设计理念、功能、输入输出规范。
*   **工具接口说明：** 集成工具的 API 文档、常见问题。

### 5.2 结构

*   **分类：** 按问题类型（数据、逻辑、LLM、性能）、智能体模块、业务场景等进行分类。
*   **标签：** 为每条知识添加关键词标签，方便搜索。
*   **关联：** 知识条目间可互相引用，形成网状结构。
*   **模版化：** 统一知识录入模版，确保信息完整性和一致性。

### 5.3 维护

*   **定期评审：** 每月组织评审会议，更新、补充、优化知识库内容。
*   **责任人机制：** 为不同模块的知识指定维护责任人。
*   **过期清理：** 定期检查并移除过时或不再适用的知识。

### 5.4 指标

*   **知识库使用率：** 团队成员查询知识库的频率。
    *   **KPI建议：** 每周人均查询次数 ≥ 5
*   **问题解决时间 (MTTR) 降低率：** 对比使用知识库前后，解决同类问题所需时间的缩短比例。
    *   **KPI建议：** MTTR 降低 ≥ 20%
*   **知识完整度：** 核心 badcase 的知识覆盖率。
    *   **KPI建议：** ≥ 90%
*   **新增知识数量：** 每月新增的知识条目数。
    *   **KPI建议：** 每月新增 ≥ 10 条

## 6. 发布门禁与迭代节奏

制定严格的发布门禁和明确的迭代节奏，确保系统高质量交付。

### 6.1 发布门禁

版本发布必须满足以下条件：

*   **冒烟测试通过率：** 100%
*   **回归测试通过率：** ≥ 98%
*   **关键 badcase 无回归：** 100%
*   **决策准确率：** 在核心评测集上达到或超过预设目标（例如 ≥ 60%）。
*   **系统性能：** 响应延迟、资源消耗等指标在可接受范围内。
*   **代码质量：** 通过静态代码检查、单元测试覆盖率等（例如单元测试覆盖率 ≥ 80%）。
*   **文档更新：** 相关设计文档、用户手册等已同步更新。

### 6.2 迭代节奏

*   **周迭代：**
    *   **周一：** 启动新一轮开发，解决上周遗留 bug。
    *   **周三：** 小版本内部测试与功能验证。
    *   **周五：** 灰度发布或部署到测试环境，进行下一轮评测集的运行和人工复核。
*   **月度大版本：**
    *   **月初：** 规划月度目标，确定重点功能和优化项。
    *   **月中：** 完成主要开发，开始集成测试和全面评测。
    *   **月末：** 满足发布门禁后，发布月度大版本。
*   **紧急修复：** 针对线上严重 badcase，可随时启动紧急修复流程，发布补丁版本。
*   **版本号规范：** 遵循 SemVer 规范 (MAJOR.MINOR.PATCH)。

## 7. v0.1 边界与后续演进

### 7.1 v0.1 边界

本框架 v0.1 版本主要聚焦于：

*   建立初步的评估维度和指标体系。
*   构建少量核心业务场景的评测集，以支持冒烟和部分回归测试。
*   明确 badcase 的发现和基本归因流程。
*   初步搭建 QA 知识库的收集和结构框架。
*   设定基本的发布门禁和周/月迭代节奏。
*   **重点：** 验证市场/社媒/新闻/基本面分析 -> 多空辩论 -> 交易计划 -> 风控辩论 -> 最终 Buy/Sell/Hold 的主链路通畅性及核心决策质量。

### 7.2 后续演进

*   **评估体系深化：** 引入更多细致的指标，例如智能体协作效率指标、风险偏好匹配度、情绪分析准确性等。
*   **评测集智能化：** 引入自动化工具，如自动生成测试用例、自动标注等，扩大评测集规模。
*   **对抗性评测：** 引入更高级的对抗性测试，发现系统潜在漏洞。
*   **A/B 测试机制：** 针对不同智能体策略、Prompt 版本进行线上 A/B 测试，量化优化效果。
*   **在线学习与自适应：** 探索系统在生产环境中持续学习和适应市场变化的能力。
*   **知识库智能化：** 引入 AI 辅助知识检索、自动问答等功能。
*   **安全与伦理：** 增加对系统决策的公平性、透明性、合规性评估。
*   **多维度可视化报告：** 自动生成更丰富、交互性更强的评测报告。
*   **更精细化的风控机制：** 深入评估 `risk_profile` 对系统行为和展示层的精确影响。
*   **压力测试与灾备演练：** 模拟极端市场条件，验证系统在重压下的稳定性和恢复能力。
