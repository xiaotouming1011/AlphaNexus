# AlphaNexus 投研系统 Gemini 版评测与迭代框架 v0.1

---

## 1. 目标与范围

### 1.1 目标
本框架旨在为基于 Gemini 模型的 AlphaNexus 多智能体投研系统，建立一套标准化、可量化的评测与迭代体系。核心目标包括：
- **质量量化**：科学评估系统在投研分析各环节的输出质量。
- **能力提升**：系统性地发现、定位并修复能力短板，提升决策分析的准确性、深度与效率。
- **迭代加速**：建立高效的 Badcase 闭环与回归测试机制，确保系统稳定性的前提下，加速产品迭代。
- **知识沉淀**：构建可复用的评测集与 QA 知识库，作为团队的核心资产。

### 1.2 范围
v0.1 版本的评测范围聚焦于 AlphaNexus 的核心分析链路（Main Chain），覆盖以下关键智能体（Agent）集群：
- **信息搜集与分析智能体 (Data Analysis Agent)**：对市场、社媒、新闻、基本面等数据的抓取、清洗与初步分析能力。
- **多空辩论智能体 (Bull-Bear Debate Agent)**：基于初步分析，从正反两方面进行深度论证的能力。
- **交易计划生成智能体 (Trading Plan Agent)**：结合辩论结果，生成具体、可执行的交易策略的能力。
- **风控辩论智能体 (Risk-Control Debate Agent)**：对交易计划进行压力测试与风险点辩论的能力。
- **最终决策合成智能体 (Final Decision Agent)**：整合所有信息，输出最终 Buy/Sell/Hold 决策及置信度的能力。

## 2. 评估体系

评估采用主观与客观维度相结合的方式，并设定权重以计算综合得分。

### 2.1 主观维度（由领域专家评测）
采用 1-5 分制（1: 完全不可用, 2: 存在严重缺陷, 3: 基本可用, 4: 表现良好, 5: 超出预期）。

| 维度 | 评估要点 | 权重 |
| :--- | :--- | :--- |
| **逻辑连贯性 (Coherence)** | - 各 Agent 间的观点传递是否流畅、无矛盾。<br>- 最终决策是否与前序分析逻辑一致。 | 25% |
| **论证深度 (Depth)** | - 分析是否停留在表面，还是挖掘了深层因果。<br>- 多空辩论是否触及关键驱动因素。 | 25% |
| **信息完整性 (Completeness)** | - 是否覆盖了影响决策的关键信息点（财报、宏观、行业、突发新闻）。<br>- `risk_profile` 是否有效影响了新闻筛选与展示。 | 20% |
| **可解释性 (Explainability)** | - 关键结论是否给出了明确的论据支撑。<br>- 来源追溯（Source Tracing）功能是否准确、有效。 | 15% |
| **洞察力 (Insightfulness)** | - 是否提出了非共识但逻辑合理的观点。<br>- 是否发现了数据间的隐藏关联。 | 15% |

### 2.2 客观维度（自动化或半自动化评测）

| 维度 | 指标与计算方式 | KPI 建议值 | 权重 |
| :--- | :--- | :--- | :--- |
| **事实准确性 (Factual Accuracy)** | - **幻觉率 (Hallucination Rate)**: `(出现事实错误的断言数 / 总断言数) * 100%`。<br>- 关键数据（如 EPS, P/E）与 Ground Truth 的偏差。 | < 5% | 35% |
| **来源追溯率 (Source Tracing Rate)** | - `(成功追溯到有效来源的断言数 / 要求追溯的总断言数) * 100%`。 | > 90% | 25% |
| **响应时效性 (Latency)** | - **端到端延迟 (End-to-End Latency)**: 从请求开始到最终决策输出的总时间。<br>- **SSE 首包延迟 (Time to First Token)**: 流式输出首个字符的响应时间。 | P95 < 15s<br>P95 < 1s | 20% |
| **稳定性 (Consistency)** | - **零温控一致性 (Zero-Temp Consistency)**: 相同输入在 temperature=0 时，多次运行的输出文本相似度（如 ROUGE-L）。 | > 0.95 | 10% |
| **模拟决策准确率 (Simulated Accuracy)** | - 在回测集上，系统 Buy/Sell 信号在未来 N 天（如 N=5, 20）内，与实际市场走向的一致性。 | > 60% (高于随机) | 10% |

### 2.3 综合评分

`最终得分 = 主观总分 * 60% + 客观总分 * 40%`

- **发布门禁要求**：核心回归集（P1）最终得分不低于 **3.8 / 5.0**。

## 3. 评测集建设与版本化

建立分层、版本化的评测集是保证评测一致性与可追溯性的基础。

### 3.1 数据获取 (Acquisition)
- **历史经典案例 (Golden Set)**：选取过去 5 年内重大市场事件，如特定公司的财报发布、并购、黑天鹅事件（如 SVB 破产）、行业政策剧变等。
- **线上高优采样 (Live Sampling)**：从线上实际运行的标的中，每周采样 Top 10 热度标的作为评测对象。
- **对抗性案例 (Adversarial Set)**：由专家构造，包含信息矛盾、数据缺失、多重利好/利空交织的复杂场景。

### 3.2 清洗与标注 (Cleaning & Annotation)
- **标准化**：将不同来源的数据（新闻、财报、社媒）格式化为统一的 JSON 结构。
- **标注**：对每个案例，由 2 位领域专家进行交叉标注，包括：
    - **关键信息点 (Key Info)**：该案例必须考虑到的核心事实和数据。
    - **预期分析路径 (Expected Path)**：合理的分析思路和逻辑链条。
    - **标准答案 (Ground Truth)**：当时的共识观点、最终的市场反应等。

### 3.3 分层管理 (Layering)
- **P0 - 冒烟集 (Smoke Test Set)**：~20个案例。覆盖主要资产类别（美股、A股、加密货币）和核心分析场景，用于 CI/CD 快速验证。目标在 5 分钟内运行完毕。
- **P1 - 核心回归集 (Regression Set)**：~100-200个案例。覆盖所有主要功能点和典型的用户场景，用于版本发布前的完整回归测试。
- **P2 - 全量评测集 (Full Set)**：~500+案例。包含所有历史、采样和对抗性案例，用于季度性的深度评测和模型选型。

### 3.4 更新与版本化 (Update & Versioning)
- **更新机制**：
    - 每周：将线上发现的高价值 Badcase 补充进 P1/P2。
    - 每月：新增近期发生的重大市场事件。
- **版本化**：评测集与代码库解耦，独立版本化管理（如 Git LFS 或 DVC）。每次评测报告需注明所使用的评测集版本号（如 `AlphaNexus_TestSet_v1.1`）。

## 4. 线上 Badcase 运营闭环

### 4.1 发现 (Discovery)
- **用户反馈**：在 UI 界面设置明确的“👍/👎”和“一键反馈”按钮。
- **内部巡检**：产品与研究团队每日进行“Dogfooding”，主动发现问题。
- **自动化监控**：设置关键指标告警，如幻觉率突增、来源追溯率骤降、P99 延迟超阈值等。

### 4.2 归因 (Attribution)
建立标准化的 Badcase Ticket 模板，强制归因到以下至少一类：
- **模型层 (Model Layer)**：Gemini 自身能力问题（如事实错误、幻觉）。
- **提示工程层 (Prompt Layer)**：Prompt 设计不佳，指令模糊或存在误导。
- **智能体协同层 (Agent Orchestration Layer)**：Agent 间通信失败、逻辑流转错误。
- **数据/工具层 (Data/Tool Layer)**：外部 API 失败、数据源质量低下或延迟。
- **用户意图理解层 (Intent Layer)**：未能准确理解用户的查询目标。

### 4.3 修复 (Fix)
- **热修复 (Hotfix)**：针对 Prompt、数据源黑名单等问题，可在 24 小时内上线。
- **常规修复 (Regular Fix)**：涉及 Agent 逻辑、工作流调整，进入下个迭代周期。
- **模型优化 (Model Finetune)**：积累足够多的高质量 Badcase 后，进行模型的 SFT（Supervised Fine-Tuning）。

### 4.4 回归 (Regression)
- **自动化**：修复后的 Badcase 必须被改编为自动化评测 case，并纳入 P1 核心回归集。
- **验证**：在版本发布前，确保所有已修复的 Badcase 在新版本中不再复现。

## 5. QA 知识库体系

### 5.1 内容收集
- **评测案例库**：收录所有 P0/P1/P2 评测集案例，包含标准输入、预期输出、专家点评。
- **Badcase 库**：收录所有线上发现的 Badcase，包含问题现象、输出快照、归因分析、修复方案。
- **优质案例库 (Goodcase)**：收录表现优异的分析案例，作为未来 Prompt 优化的参考和团队演示材料。

### 5.2 知识结构
建议使用 Confluence 或类似文档系统，按以下结构组织：
- `AlphaNexus QA`
    - `评测集 (Test Sets)`
        - `v1.0`
        - `v1.1`
    - `案例库 (Case Library)`
        - `Badcases`
            - `按失败类型分类（如幻觉、逻辑错误）`
        - `Goodcases`
            - `按分析主题分类（如财报分析、宏观分析）`
    - `评测指南与规范 (Guidelines)`

### 5.3 维护机制
- **责任人**：指定 QA 负责人，每周同步更新知识库。
- **流程集成**：将知识库更新作为 Badcase Ticket 关闭的最后一个步骤。

### 5.4 关键指标
- **知识库覆盖率**：`已归档案例数 / 已发现总案例数`，目标 > 95%。
- **案例复用率**：在进行模型/Prompt 优化时，从知识库中直接调用的案例比例。

## 6. 发布门禁与迭代节奏

### 6.1 发布门禁 (Release Gate)
任何版本发布（含热修复）必须满足以下条件：
1.  **单元/集成测试通过率**：100%。
2.  **P0 冒烟集评测通过率**：100%。
3.  **P1 核心回归集评测**：
    - 综合得分 ≥ 3.8。
    - 关键客观指标（事实准确性、来源追溯率）无劣化。
    - 无新增的 P0/P1 级别 Regression Badcase。
4.  **线上严重 Badcase 关闭率**：100%。

### 6.2 迭代节奏
- **日级别 (Daily)**：CI/CD 触发 P0 冒烟测试。
- **周级别 (Weekly)**：进行 Prompt 微调、Bug修复、数据源更新等。发布前执行完整的 P1 回归测试。
- **月/双周级别 (Monthly/Bi-weekly)**：进行 Agent 逻辑、功能模块等较大更新。发布前执行完整的 P1 回归测试。
- **版本/季度级别 (Version/Quarterly)**：进行模型基座更换（如升级 Gemini 新版本）、重构等重大变更。执行 P0/P1/P2 全量评测，并生成详细的横向对比报告。

## 7. v0.1 边界与后续演进

### 7.1 v0.1 边界
- **评测重心**：侧重于分析内容的文本质量（逻辑、深度、准确性），暂不将最终决策的绝对盈亏作为核心评估指标。
- **自动化程度**：主观评测依赖专家，客观评测为半自动化脚本。
- **资产范围**：评测集优先覆盖高流动性的美股、主流加密货币。
- **个性化**：`risk_profile` 的评测仅限于其对信息展示层的影响，暂不深入评估其对决策风格的根本性改变。

### 7.2 后续演进
- **评测自动化 (Evaluation Automation)**：探索引入“AI-as-Judge”机制，使用更高阶的 LLM（或多个 LLM 投票）对主观维度进行初步评分，提高评测效率。
- **闭环回测集成 (Closed-loop Backtesting)**：将系统输出的交易计划程序化，接入回测框架，实现基于 PnL、夏普比率等金融指标的端到端评测。
- **多模态能力评测 (Multi-modal Evaluation)**：当系统集成图表、K线图等视觉分析能力时，扩展评测体系，覆盖对视觉信息的理解与分析能力。
- **个性化评测扩展 (Personalization Evaluation)**：设计专门的评测集，验证系统能否根据不同的 `risk_profile`（如保守型 vs. 激进型）生成风格迥异且逻辑自洽的分析与决策。
